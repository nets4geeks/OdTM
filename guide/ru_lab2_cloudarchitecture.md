# Лабораторная работа 2: Моделирование облачных архитектурных угроз

### Введение

Эффективное использование шаблонов угроз требует некоторого понимания аспектов безопасности и наличие небольшого опыта архитектурного проектирования.
Для неподготовленных пользователей предлагается набор архитектурных примитивов, изучив которые заранее, можно лучше понять особенности архитектуры конкретной системы.

В этой лабораторной работе рассматриваются три основных архитектурных примитива облачных систем, а [каталог шаблонов угроз](https://nets4geeks.github.io/acctp/catalog/) содержит соответствующие перечни угроз.

### Первый примитив: Простое облачное приложение

Простейший вариант архитектуры может быть представлен взаимодействием удаленного пользователя (Remote User) с облачным приложением (Cloud Application):

![lab2_primitive1](pics/lab2_primitive1.png)

С точки зрения сетевых технологий удаленный пользователь является клиентом, а облачное приложение - сервером.
Некоторый прикладной протокол (например, HTTPS) поверх стека TCP/IP используется для передачи данных.

Облачное приложение может быть источником угроз для сущностей, выступающих в качестве клиентов (удаленный пользователь, внешний сервис).
Примером угрозы данного типа (группа **ABxx** в каталоге шаблонов угроз) является [AB01 Сбой облачного приложения (Failure Of Cloud Application)](https://nets4geeks.github.io/acctp/catalog/threatAB01_FailureOfCloudApplication.html).
Группа **ACxx** представляет угрозы, специфичные только для удаленных пользователей, например [AC01 Заражение вирусом (вредоносным ПО) от облачного приложения (Malware From Cloud Application)](https://nets4geeks.github.io/acctp/catalog/threatAC01_MalwareFromCloudApplication.html).

Клиенты могут угрожать облачному приложению.
Группа **ADxx** содержит общие угрозы облачному приложению от удаленных пользователей, внешних сервисов и приложений,
например [AD01 Нарушениея аутентификации (Broken Authentication)](https://nets4geeks.github.io/acctp/catalog/threatAD01_BrokenAuthentication.html).
Группа **AExx** включает угрозы, создаваемые только удаленным пользователем, например [AE01 Атака "Распределенный отказ в обслуживании" на облачное приложение (DDoS Attack To Cloud Application)](https://nets4geeks.github.io/acctp/catalog/threatAE01_DDoSAttackToCloudApplication.html).

Возможная модель угроз для первого примитива показана в [примере 1](models/lab2_primitive1_modelled.json).

### Второй примитив: Взаимодействие облачных приложений

Облачное приложение может иметь несколько компонентов, например, интерфейс пользователя (User Interface - UI) и прикладной программный интерфейс (Application Programming Interface - API).
Компоненты приложения влияют друг на друга с точки зрения информационной безопасности.

![lab2_primitive2](pics/lab2_primitive2.png)

Шаблоны угроз, относящиеся ко второму примитиву, находятся в группах **ABxx** and **ADxx** (см. примеры выше).

Возможная модель угроз для второго примитива показана в [примере 2](models/lab2_primitive2_modelled.json).

### Третий примитив: Взаимодействие с внешними сервисами

Облачное приложение может использовать сервисы, рассматриваемые как "внешние", т.е. предоставляющие услуги без соглашения между сервисом и потребителем,
а также те, управление которыми ограничено (сетевой поток Cloud Application - External Service).

![lab2_primitive3](pics/lab2_primitive3a.png)

Этот случай (использование внешнего сервиса) описан в группе шаблонов угроз **AAxx**.
Пример: [AA05 Подмена удаленного сервиса (Spoofing Of Remote Service)](https://nets4geeks.github.io/acctp/catalog/threatAA05_SpoofingOfRemoteService.html).

Возможен также случай использования облачного приложения внешним сервисом (сетевой поток External Service - Cloud Application).

![lab2_primitive3](pics/lab2_primitive3b.png)

Использование облачных приложений с точки зрения безопасности рассмотрено в группах **ABxx** и **ADxx**, упомянутых выше.

Возможная модель угроз для третьего примитива показана в [примере 3](models/lab2_primitive3_modelled.json).
